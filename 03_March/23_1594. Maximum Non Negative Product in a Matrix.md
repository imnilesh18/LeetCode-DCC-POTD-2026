## 📜 Problem Statement

You are given a `m x n` matrix `grid`. Initially, you are located at the top-left corner `(0, 0)`, and in each step, you can only move **right** or **down** in the matrix.

Among all possible paths starting from the top-left corner `(0, 0)` and ending in the bottom-right corner `(m - 1, n - 1)`, find the path with the **maximum non-negative product**. The product of a path is the product of all integers in the grid cells visited along the path.

Return the **maximum non-negative product modulo** `10^9 + 7`. If the maximum product is **negative**, return `-1`.

> **Notice** that the modulo is performed after getting the maximum product.

---

## 🚫 Constraints

- `m == grid.length`
- `n == grid[i].length`
- `1 <= m, n <= 15`
- `-4 <= grid[i][j] <= 4`

## 🧠 Key Insights & Explanation

If the grid contained only positive numbers, this would be a simple standard pathfinding Dynamic Programming problem where we just pick the `max` product at every step. However, the presence of **negative numbers** changes the entire dynamic.

**Why does this happen?**
Imagine you arrive at a cell `(i, j)` with a value of `-2`. 
- If the path from the top brought a max product of `12`, multiplying yields `-24`. 
- But if the path from the left brought a minimum (highly negative) product of `-8`, multiplying `-8 * -2` yields `16`!

Thus, a negative running product can instantly become a massive positive product when multiplied by another negative value. 

Because of this, we **must track both the maximum and minimum products** at every single cell. At each cell `(i, j)`, the new maximum and minimum can be formed by multiplying the current cell's value with either the `max` or the `min` coming from the cell above or the cell to the left. Finally, if the maximum product reaching the bottom-right cell is still negative, we return `-1`. Otherwise, we apply the modulo.

---

## 🚀 Approach

We can solve this problem efficiently using Dynamic Programming. Two approaches are demonstrated below:

1. **Top-Down (Recursion + Memoization):** We compute paths starting from `(0,0)` to `(m-1, n-1)`. At each state, we store `{maxProduct, minProduct}` to prevent duplicate recursive calculations.
2. **Bottom-Up (Iterative DP):** We maintain a 2D state matrix `t` where `t[i][j]` holds the pair `{max, min}`. We pre-fill the first row and the first column, then iteratively fill the rest of the grid using combinations of above and left states.

## 💻 Solution Code

### Approach 1: Recursive Memoization

```cpp
// Intuition: Since negative numbers exist, a highly negative minimum product can become a large positive product when multiplied by a negative cell. Thus, tracking both max and min products is strictly required.
// Approach: Recursively explore the grid (down and right) to find the max and min possible products. We use a 2D vector of pairs to memoize states, ensuring each cell is computed exactly once. If the final max is negative, we yield -1.
// Time Complexity: O(m * n) - Each cell is visited and computed exactly once due to 2D memoization.
// Space Complexity: O(m * n) - O(m * n) space for the DP table and up to O(m + n) for recursive stack space.

class Solution {
public:
    int m, n;
    typedef long long ll;
    int MOD = 1e9+7;

    vector<vector<pair<ll, ll>>> t;
    
    pair<ll, ll> solve(int i, int j, vector<vector<int>>& grid) {
        // Base case: destination reached
        if(i == m-1 && j == n-1) {
            return {grid[i][j], grid[i][j]};
        }

        ll maxVal = LLONG_MIN;
        ll minVal = LLONG_MAX;

        // Return memoized result if already calculated
        if(t[i][j] != make_pair(LLONG_MIN, LLONG_MAX)) {
            return t[i][j];
        }

        // Down
        if(i+1 < m) {
            auto [downMax, downMin] = solve(i+1, j, grid);
            maxVal = max({maxVal, grid[i][j] * downMax, grid[i][j] * downMin});
            minVal = min({minVal, grid[i][j] * downMax, grid[i][j] * downMin});
        }

        // Right
        if(j+1 < n) {
            auto [rightMax, rightMin] = solve(i, j+1, grid);
            maxVal = max({maxVal, grid[i][j] * rightMax, grid[i][j] * rightMin});
            minVal = min({minVal, grid[i][j] * rightMax, grid[i][j] * rightMin});
        }

        // Save states to avoid redundant calculations
        return t[i][j] = {maxVal, minVal};
    }
    int maxProductPath(vector<vector<int>>& grid) {
        m = grid.size();
        n = grid[0].size();
        
        t = vector<vector<pair<ll, ll>>>(m, vector<pair<ll, ll>>(n, {LLONG_MIN, LLONG_MAX}));
        auto [maxProd, minProd] = solve(0, 0, grid);

        return maxProd < 0 ? -1 : maxProd % MOD;
    }
};

/*
*
* Dry Run
* Grid = [[1, 3],
* [0, -4]]
* solve(0, 0): Try moving Right to (0,1) and Down to (1,0).
* solve(0, 1): Reaches (0,1), tries moving down to (1,1). Base case (1,1) returns {-4, -4}.
* Multiplies 3 * {-4, -4} = {-12, -12}. Returns {-12, -12} up.
* solve(1, 0): Reaches (1,0), tries moving right to (1,1). Returns {-4, -4}.
* Multiplies 0 * {-4, -4} = {0, 0}. Returns {0, 0} up.
* Back to (0,0): 
* From Right: 1 * {-12, -12} -> max: -12, min: -12
* From Down:  1 * {0, 0} -> max: 0, min: 0
* Comparing these gives maxProd = 0.
* 0 >= 0, so return 0 % MOD = 0.
*
*/
````

### Approach 2: Bottom-Up Dynamic Programming (Optimized)

```cpp
// Intuition: We can eliminate recursion overhead by computing values from the top-left cell to the bottom-right systematically.
// Approach: We build a DP matrix `t` where `t[i][j]` holds the pair {maxVal, minVal} up to that cell. After initializing `t[0][0]`, we prefill the 0th row and 0th column. For remaining cells, we generate possible values coming from the Top and Left cells and update both the global max and min reaching `(i, j)`.
// Time Complexity: O(m * n) - Nested loops to traverse every element of the matrix once.
// Space Complexity: O(m * n) - Space used by the DP table holding pair values.

class Solution {
public:
    typedef long long ll;
    int MOD = 1e9+7;

    int maxProductPath(vector<vector<int>>& grid) {
        int m = grid.size();
        int n = grid[0].size();
        
        // bottom up dp lena padega
        vector<vector<pair<ll, ll>>> t(m, vector<pair<ll, ll>>(n));
        // t[i][j] = {maxVal, minVal} to reach from (0, 0) to (i, j)

        t[0][0] = {grid[0][0], grid[0][0]};

        // filling the first row
        for(int j = 1; j < n; j++) {
            t[0][j].first  = t[0][j-1].first  * grid[0][j]; // maxVal
            t[0][j].second = t[0][j-1].second * grid[0][j]; // minVal
        }

        // filling the first column
        for(int i = 1; i < m; i++) {
            t[i][0].first  = t[i-1][0].first  * grid[i][0]; // max Val
            t[i][0].second = t[i-1][0].second * grid[i][0]; // min val
        }

        for(int i = 1; i < m; i++) {
            for(int j = 1; j < n; j++) {
                ll upMax = t[i-1][j].first;
                ll upMin = t[i-1][j].second;

                ll leftMax = t[i][j-1].first;
                ll leftMin = t[i][j-1].second;

                t[i][j].first  = max({upMax * grid[i][j], upMin * grid[i][j], leftMax * grid[i][j], leftMin * grid[i][j]});
                t[i][j].second = min({upMax * grid[i][j], upMin * grid[i][j], leftMax * grid[i][j], leftMin * grid[i][j]});
            }
        }

        auto [maxProd, minProd] = t[m-1][n-1];

        return maxProd < 0 ? -1 : maxProd % MOD;
    }
};

/*
*
* Dry Run
* Grid = [[1, -2, 1],
* [1, -2, 1],
* [3, -4, 1]]
* 1. Initialize t[0][0] = {1, 1}
* 2. First Row: t[0][1] = {-2, -2}, t[0][2] = {-2, -2}
* 3. First Col: t[1][0] = {1, 1}, t[2][0] = {3, 3}
* 4. For (1,1) grid value -2:
* - Up is {-2, -2} -> Multiplied by -2 -> {4, 4}
* - Left is {1, 1} -> Multiplied by -2 -> {-2, -2}
* - Maximum of all = 4, Minimum of all = -2
* - t[1][1] = {4, -2}
* 5. Repeating this logic populates the entire grid evaluating negative inversion.
* 6. At (2,2), we retrieve the maxProd of 8. Since 8 >= 0, return 8.
*
*/
```

## 📝 Notes

  - **Warning on Modulo Constraints:** Pay special attention\! You should **not** perform modulo arithmetic during the intermediate cell calculations. The modulo `10^9+7` must be applied *only* to the final computed answer. Premature modulo operations will break the mathematical logic of discovering the genuine mathematical maximum/minimum.
  - Utilizing `long long` (`ll`) is crucial because repeated multiplications of boundary variables up to `15x15` dimensions can cause immense integer overflow prior to the final modulo sequence.
