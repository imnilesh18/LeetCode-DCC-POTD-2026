## 📝 Problem Statement

**Problem Link:** [Fancy Sequence on LeetCode](https://leetcode.com/problems/fancy-sequence/description/)

Write an API that generates fancy sequences using the `append`, `addAll`, and `multAll` operations.

Implement the `Fancy` class:
- `Fancy()` Initializes the object with an empty sequence.
- `void append(val)` Appends an integer `val` to the end of the sequence.
- `void addAll(inc)` Increments all existing values in the sequence by an integer `inc`.
- `void multAll(m)` Multiplies all existing values in the sequence by an integer `m`.
- `int getIndex(idx)` Gets the current value at index `idx` (0-indexed) of the sequence **modulo** $10^9 + 7$. If the index is greater or equal than the length of the sequence, return `-1`.

### 🚨 Constraints
- $1 \le val, inc, m \le 100$
- $0 \le idx \le 10^5$
- At most $10^5$ calls total will be made to `append`, `addAll`, `multAll`, and `getIndex`.

## 🧠 Approach & Explanation

### The Naive Approach (Why it fails)
The brute-force way is a direct simulation: update the entire sequence array for every `addAll` or `multAll` call. However, since the sequence size can grow up to $10^5$ and there can be $10^5$ operations, the worst-case time complexity would be $O(N \times Q) \approx 10^{10}$, which will undoubtedly result in a **Time Limit Exceeded (TLE)**.

### The Optimized Approach: Lazy Evaluation & Modular Arithmetic
Instead of applying the updates to every element in the array immediately, we use **Lazy Evaluation**. We maintain global `add` and `mult` tracking variables.

1. **Global Tracking:** - Initialize `add = 0` and `mult = 1`.
   - When `addAll(inc)` is called: update `add = (add + inc) % MOD`.
   - When `multAll(m)` is called: update both `mult = (mult * m) % MOD` and `add = (add * m) % MOD` (since the multiplication affects the previously added values as well).

2. **Evaluating an Index:**
   - Instead of modifying the array elements, `getIndex(idx)` evaluates the answer on-the-fly:
   - `Final Value = (Sequence[idx] * mult + add) % MOD`

3. **Handling the Appends (The Catch!):**
   - If we simply append a new `val` to our sequence, a future `getIndex` call will apply the global `mult` and `add` to it, treating it as if it was there from the beginning.
   - To counteract this, we need to append a "normalized" value `x` such that when the current `mult` and `add` are applied to it, it reconstructs `val`.
   - **Equation:** $x \cdot mult + add = val$
   - **Solving for x:** $x = (val - add) \cdot mult^{-1} \pmod{MOD}$
   
4. **Modular Inverse (Fermat's Little Theorem):**
   - Division under modulo arithmetic isn't straightforward. We must multiply by the modular inverse.
   - Since $MOD = 10^9 + 7$ is a prime number, by **Fermat's Little Theorem**, $mult^{-1} \equiv mult^{MOD-2} \pmod{MOD}$.
   - We can efficiently compute this using **Binary Exponentiation** in $O(\log(MOD))$ time.

## 👨‍💻 Solution Code

```cpp
// Intuition: We use Lazy Evaluation to track global addition and multiplication operations. When appending a new value, we "reverse" these global operations on the value before storing it using Fermat's Little Theorem and Modular Inverse.
// Approach:
// 1. Maintain global track of 'add' (initially 0) and 'mult' (initially 1).
// 2. addAll(inc) simply updates add = (add + inc) % MOD.
// 3. multAll(m) updates both mult = (mult * m) % MOD and add = (add * m) % MOD.
// 4. getIndex(idx) returns (seq[idx] * mult + add) % MOD.
// 5. append(val) inserts a normalized value x = (val - add) * modInverse(mult) % MOD to counteract past operations. We compute the modular inverse using Binary Exponentiation.
// Time Complexity: O(log M) for append due to Binary Exponentiation, O(1) for addAll, multAll, and getIndex.
// Space Complexity: O(N) to store the sequence elements where N is the number of appends.

class Fancy {
public:
    typedef long long ll;
    ll M = 1e9+7;

    vector<ll> seq;
    ll add = 0;
    ll mult = 1;

    // Binary Exponentiation to calculate (a^b) % M efficiently
    long long power(long long a, long long b) {
        if(b == 0) return 1;

        long long half = power(a, b / 2);
        long long result = (half * half) % M;

        // If exponent is odd, multiply by base one more time
        if(b % 2 == 1) {
            result = (result * a) % M;
        }
        return result;
    }

    Fancy() {
        // Constructor initializes empty, handled by class members
    }
    
    void append(int val) {
        // Reverse operations: (val - add) * mult^-1. Add +M before modulo to handle negative values.
        long long x = ((val - add) % M + M) % M * power(mult, M - 2) % M;
        seq.push_back(x);
    }
    
    void addAll(int inc) {
        // Update global add factor
        add = (add + inc) % M;
    }
    
    void multAll(int m) {
        // Update global mult and add factors (past additions get multiplied too)
        mult = (mult * m) % M;
        add  = (add * m) % M;
    }
    
    int getIndex(int idx) {
        if(idx >= seq.size()) return -1;
        // Apply global track factors to get the actual value at the time of querying
        return (seq[idx] * mult + add) % M;
    }
};

/*
*
* Dry Run
*
* Operations:
* 1. addAll(2) -> add = 2, mult = 1
* 2. multAll(3) -> mult = 1 * 3 = 3, add = 2 * 3 = 6
* 3. append(7) -> we need to store x where x * 3 + 6 = 7.
* x = (7 - 6) * modInverse(3) % M.
* We store x in the array.
* 4. getIndex(0) -> Retrieves x from array.
* Returns (x * mult + add) % M 
* = (x * 3 + 6) % M 
* = 7. Correct! The sequence logic successfully isolates new values from past global updates.
*
*/
```

## 🔍 Key Insights

- **Lazy Evaluation:** A powerful technique allowing $O(1)$ operations on arrays where bulk updates apply universally. You only compute the exact state of an item when it is explicitly requested.
- **Modular Subtraction Safety:** Always remember to do `((val - add) % M + M) % M` when subtracting in modular arithmetic to prevent negative remainders.
- **Motivational Insight:** *“A ship in harbor is safe, but that is not what ships are built for.”* Getting out of the comfort zone of simulation/brute force and stepping into mathematical derivation is exactly how hard competitive programming problems are solved!
