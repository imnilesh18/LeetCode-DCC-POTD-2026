## 📝 Problem Statement

You are given an `m x n` integer matrix `grid`.

A **rhombus sum** is the sum of the elements that form the **border** of a regular rhombus shape in `grid`. The rhombus must have the shape of a square rotated 45 degrees with each of the corners centered in a grid cell. Below is an image of four valid rhombus shapes with the corresponding colored cells that should be included in each rhombus sum:

Note that the rhombus can have an area of 0, which is depicted by the purple rhombus in the bottom right corner.

Return *the biggest three **distinct** rhombus sums in the `grid` in **descending order***. If there are less than three distinct values, return all of them.

## 💡 Explanation & Approach

### 1. Brute Force Approach
The problem asks us to find the top three distinct rhombus sums. A common misconception is that a visually larger rhombus (or one enclosing a cell with a huge value like `200`) will strictly yield the highest sum. However, it's solely about the values on the **border** of the rhombus. Since we cannot greedily guess where the best rhombus lies, we must check every possibility.

1. Treat every cell `(r, c)` as the **center** of a potential rhombus.
2. An area of `0` is a valid rhombus. For any center `(r, c)`, a rhombus of `side = 0` yields a sum of `grid[r][c]`.
3. Expand outwards to form larger rhombuses (`side = 1, 2, ...`). For a given `side`, the coordinates of the 4 corners are:
   - **Top:** `(r - side, c)`
   - **Right:** `(r, c + side)`
   - **Bottom:** `(r + side, c)`
   - **Left:** `(r, c - side)`
4. Iterate over the boundary cells using a loop to sum up the values of the 4 edges. 
5. Keep track of the top 3 maximum distinct sums using an ordered `set`.

### 2. Optimized Prefix Sum Approach
The brute-force solution manually iterates over each diagonal edge to calculate the sum. For large rhombuses, this means redundant addition sequences spanning overlapping cells. 
We can optimize this edge traversal from $O(side)$ to $O(1)$ by precomputing **Diagonal Prefix Sums**:
- `d1` array stores prefix sums for diagonals moving from top-left to bottom-right ($\searrow$).
- `d2` array stores prefix sums for diagonals moving from top-right to bottom-left ($\swarrow$).

With these arrays, the sum of any diagonal edge can be found using subtraction. Because this prefix sum logic incorporates the corners in two adjacent edges, each of the 4 corners will be **double-counted**. To fix this, simply subtract the four corner values from the aggregated sum once at the end.

## 💻 Solution Code

### Approach 1: Brute Force

```cpp
// Intuition: Since we don't know which rhombus yields the max sum, we must explore every valid rhombus centered at each cell. We iterate through all cells and expand outwards to form rhombuses of all possible sizes.
// Approach: 
// 1. Iterate through each cell (r, c) treating it as the center.
// 2. For each center, iterate through all valid 'side' lengths. Side 0 is just the cell itself.
// 3. For side > 0, calculate the sum along the 4 diagonal edges using a loop.
// 4. Use an ordered set to maintain the top 3 distinct maximum sums. Maintain size <= 3 by erasing the smallest element.
// Time Complexity: O(m * n * min(m,n)^2), where m is rows and n is columns. For each cell, the side length goes up to min(m,n)/2, and edge traversal takes O(side).
// Space Complexity: O(1) auxiliary space (ignoring the small set of size 3).

class Solution {
public:
    vector<int> getBiggestThree(vector<vector<int>>& grid) {
        int m = grid.size();
        int n = grid[0].size();
        set<int> st; // Stores up to 3 distinct maximum sums

        // Helper lambda to maintain top 3 distinct elements
        auto addToSet = [&](int val) {
            st.insert(val);
            if(st.size() > 3) {
                st.erase(begin(st)); // Remove smallest element
            }
        };

        // Iterate through every cell acting as a center
        for(int r = 0; r < m; r++) {
            for(int c = 0; c < n; c++) {

                addToSet(grid[r][c]); // every cell is valid for rohmbus of side = 0

                // Expand side lengths while staying within grid bounds
                for(int side = 1; r - side >= 0 && r + side < m && c - side >= 0 && c + side < n; side++) {

                    int sum = 0;

                    // Traverse the 4 edges of the rhombus
                    for(int k = 0; k < side; k++) {
                        sum += grid[r - side + k][c + k]; // top to right corner
                        sum += grid[r + k][c + side - k]; // right to bottom corner
                        sum += grid[r + side - k][c - k]; // bottom to left corner
                        sum += grid[r - k][c - side + k]; // left to top corner
                    }

                    addToSet(sum); // Update result set
                }
            }
        }

        // Return elements in descending order
        return vector<int>(rbegin(st), rend(st));
    }
};

/*
*
* Dry Run
* Let grid = [[1,2,3],[4,5,6],[7,8,9]]
* m=3, n=3. Set 'st' tracks max sums.
* r=1, c=1 (center 5):
* side=0: add 5. st={5}
* side=1: valid bounds.
* k=0: sum += grid[0][1](2) + grid[1][2](6) + grid[2][1](8) + grid[1][0](4) = 20.
* add 20. st={5, 20}
* End of iterations. Other cells will add 1,2,3,4,6,7,8,9 (side 0).
* Top 3 are 20, 9, 8. Result returned is [20, 9, 8].
*
*/

```

### Approach 2: Optimized using Diagonal Prefix Sums

```cpp
// Intuition: Calculating the sum of edges using a loop causes repeated work. We can optimize edge summation to O(1) by precomputing diagonal prefix sums in both directions (\ and /).
// Approach:
// 1. Precompute `d1` for top-left to bottom-right diagonals and `d2` for top-right to bottom-left diagonals.
// 2. Iterate through all cells as centers and all valid side lengths.
// 3. For a given side, define the 4 corners: top, right, bottom, left.
// 4. Use `d1` and `d2` to get the sum of the 4 edges in O(1) time.
// 5. Since the 4 corners are included in two adjacent edges during the prefix sum logic, subtract them once to fix double-counting.
// 6. Use a set to maintain the top 3 distinct sums.
// Time Complexity: O(m * n * min(m, n)). Precomputing takes O(m*n). Finding valid rhombuses takes O(m*n * min(m,n)) because edge sums are now O(1).
// Space Complexity: O(m * n) to store the prefix sum arrays d1 and d2.

class Solution {
public:
    vector<int> getBiggestThree(vector<vector<int>>& grid) {

        int m = grid.size(), n = grid[0].size();

        // left to right diagonals prefix (\)
        vector<vector<int>> d1(m, vector<int>(n));

        // right to left diagonals prefix (/)
        vector<vector<int>> d2(m, vector<int>(n));

        // building d1
        for(int i = 0; i < m; i++){
            for(int j = 0; j < n; j++){
                d1[i][j] = grid[i][j];
                if(i > 0 && j > 0)
                    d1[i][j] += d1[i-1][j-1];
            }
        }

        // building d2
        for(int i = 0; i < m; i++){
            for(int j = n-1; j >= 0; j--){
                d2[i][j] = grid[i][j];
                if(i > 0 && j+1 < n)
                    d2[i][j] += d2[i-1][j+1];
            }
        }

        set<int> st;

        // Helper lambda to keep top 3 distinct elements
        auto addToSet = [&](int val){
            st.insert(val);
            if(st.size() > 3)
                st.erase(st.begin()); // Drop smallest element
        };

        for(int r = 0; r < m; r++){
            for(int c = 0; c < n; c++){

                // rhombus with side = 0
                addToSet(grid[r][c]);

                // Expand side while within boundaries
                for(int side = 1; r-side >= 0 && r+side < m && c-side >= 0 && c+side < n; side++) {
                    int sum = 0;

                    // Define the four corners
                    int top_r = r-side, top_c = c;
                    int right_r = r, right_c = c+side;
                    int bottom_r = r+side, bottom_c = c;
                    int left_r = r, left_c = c-side;

                    // top to right corner (\ diagonal)
                    sum += d1[right_r][right_c];
                    if(top_r-1 >= 0 && top_c-1 >= 0)
                        sum -= d1[top_r-1][top_c-1];

                    // right to bottom corner (/ diagonal)
                    sum += d2[bottom_r][bottom_c];
                    if(right_r-1 >= 0 && right_c+1 < n)
                        sum -= d2[right_r-1][right_c+1];

                    // bottom to left corner (\ diagonal)
                    sum += d1[bottom_r][bottom_c];
                    if(left_r-1 >= 0 && left_c-1 >= 0)
                        sum -= d1[left_r-1][left_c-1];

                    // left to top corner (/ diagonal)
                    sum += d2[left_r][left_c];
                    if(top_r-1 >= 0 && top_c+1 < n)
                        sum -= d2[top_r-1][top_c+1];

                    // remove corners counted twice
                    sum -= grid[top_r][top_c];
                    sum -= grid[right_r][right_c];
                    sum -= grid[bottom_r][bottom_c];
                    sum -= grid[left_r][left_c];

                    addToSet(sum); // Update result set
                }
            }
        }

        // Return sorted in descending order
        return vector<int>(st.rbegin(), st.rend());
    }
};

/*
*
* Dry Run
* Let grid = [[1,2,3],[4,5,6],[7,8,9]]
* d1 ( \ diag ):
* [1, 2, 3]
* [4, 6, 8]
* [7, 12,15]
* d2 ( / diag ):
* [1, 2, 3]
* [6, 8, 6]
* [15,14, 9]
* r=1, c=1 (center 5), side=1.
* Corners: top(0,1)=2, right(1,2)=6, bottom(2,1)=8, left(1,0)=4
* Edge 1 (top->right): d1[1][2] - d1[-1][0] = 8 - 0 = 8
* Edge 2 (right->bottom): d2[2][1] - d2[0][3] = 14 - 0 = 14
* Edge 3 (bottom->left): d1[2][1] - d1[0][-1] = 12 - 0 = 12
* Edge 4 (left->top): d2[1][0] - d2[-1][2] = 6 - 0 = 6
* Total sum = 8+14+12+6 = 40.
* Corners subtracted: 40 - (2+6+8+4) = 40 - 20 = 20.
* Sum added = 20.
* Maximum 3 sums eventually evaluate to [20, 9, 8].
*
*/

```

## 📌 Key Insights & Notes

* **Area of Size 0:** The problem requires considering a `1x1` single cell as a "rhombus of area 0" which implies the minimum possible sum is just `grid[i][j]`.
* **Set Data Structure:** The constraint requires returning the *distinct* values in descending order. A standard `std::set` in C++ stores values distinctly and in sorted (ascending) order. Keeping the size strictly to `<= 3` by calling `erase(st.begin())` provides massive efficiency as it drops the smallest element continuously. Reverse iterating maps ascending to descending at the end.
* **Prefix Sum over Diagonals:** Understanding how to traverse diagonally (`d1` uses `[i-1][j-1]` while `d2` uses `[i-1][j+1]`) is an immensely valuable technique that extends beyond this specific problem to any pattern requiring rotated grid queries.
