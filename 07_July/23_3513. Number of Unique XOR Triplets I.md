## 📝 Problem Statement

You are given an integer array `nums` of length `n`, where `nums` is a permutation of the numbers in the range `[1, n]`.

A **XOR triplet** is defined as the bitwise XOR of three elements `nums[i] XOR nums[j] XOR nums[k]` where `i <= j <= k`.

Return the number of unique XOR triplet values from all possible triplets `(i, j, k)`.

---

## 🚧 Constraints

* `1 <= n == nums.length <= 10^5`
* `1 <= nums[i] <= n`
* `nums` is a permutation of integers from `1` to `n`.

## 🧠 Approach & Explanation

The problem asks us to find the number of unique XOR values generated from all possible triplets $(i, j, k)$ such that $i \le j \le k$. Since $n \le 10^5$, calculating all triplets iteratively will take $O(n^3)$ time, resulting in a Time Limit Exceeded (TLE) error. We need a more optimal approach, ideally $O(n)$ or $O(\log n)$.

**Observations on XOR and Permutations:**
1. The fact that the array is a permutation of `1` to `n` is slightly misleading because the actual sequence of the numbers doesn't change the unique XOR values we can form. `nums = [1, 2]` yields the exact same unique XORs as `nums = [2, 1]`. The actual values present (`1` through `n`) are what matter.
2. For $n = 1$, the only triplet is `(1, 1, 1)` yielding `1`. So the answer is `1`.
3. For $n = 2$, the triplets yield `{1, 2}`. So the answer is `2`.
4. For $n \ge 3$, the unique values start to include `0` (e.g., `1 ^ 2 ^ 3 = 0`). 
5. XOR operations will never "create" a higher bit than the ones already present in the maximum number within our set. For instance, if $n = 5$ (which requires 3 bits to represent: `101` in binary), no combination of XOR operations between numbers $\le 5$ will flip a 4th bit (like $2^3 = 8$). The absolute maximum value you can achieve is by setting all 3 bits to `1` (which gives $7$).
6. Because we can reuse numbers (due to $i \le j \le k$), we can actually form **every single number** from $0$ up to that maximum theoretical bound. Thus, the number of unique values is strictly bounded by the number of bits required to represent $n$. 

**Conclusion:**
For any $n \ge 3$, the total number of unique XOR values is exactly the next power of 2 that is strictly greater than $n$. 
* If $n = 3$, the next power of 2 is `4`. (Yields `0, 1, 2, 3` -> 4 unique)
* If $n = 4$, the next power of 2 is `8`. (Yields `0` through `7` -> 8 unique)
* If $n = 5$, the next power of 2 is `8`.

## 💻 Brute-force Solution (C++)

<details>
<summary>Click to view Brute-force Code (O(N^3) - TLE)</summary>

```cpp
// Intuition: Check every possible triplet combinations (i, j, k) and use a HashSet to keep track of uniquely generated XOR sums.
// Approach: Iterate via three nested loops maintaining the constraint i <= j <= k. Compute the XOR and insert it into a set. Return the size of the set.
// Time Complexity: O(n^3) - Evaluating all combinations will result in Time Limit Exceeded (TLE).
// Space Complexity: O(n) - The unordered_set stores the unique XOR values.

class Solution {
public:
    int uniqueXorTriplets(vector<int>& nums) {
        int n = nums.size();
        unordered_set<int> uniqueValues;
        
        // Check all valid (i, j, k) combinations
        for (int i = 0; i < n; ++i) {
            for (int j = i; j < n; ++j) {
                for (int k = j; k < n; ++k) {
                    uniqueValues.insert(nums[i] ^ nums[j] ^ nums[k]);
                }
            }
        }
        
        return uniqueValues.size();
    }
};

/*
*
* Dry Run
* 1. nums = [1, 2], n = 2
* 2. i=0, j=0, k=0 -> 1^1^1 = 1 (Set: {1})
* 3. i=0, j=0, k=1 -> 1^1^2 = 2 (Set: {1, 2})
* 4. i=0, j=1, k=1 -> 1^2^2 = 1 (Set: {1, 2})
* 5. i=1, j=1, k=1 -> 2^2^2 = 2 (Set: {1, 2})
* 6. Returns Set Size: 2
*
*/
```

## 💻 Optimized Solution (C++)

```cpp
// Intuition: For n >= 3, the unique XOR values generated cover all integers from 0 up to the maximum possible number formed by setting all bits up to the most significant bit of n. This equates to finding the nearest power of 2 strictly greater than n.
// Approach: Check if n is 1 or 2, and return n directly. For larger numbers, iterate by left-shifting an answer variable (starting from 1) until it exceeds n.
// Time Complexity: O(log2(n)) - The while loop runs equal to the number of bits in n.
// Space Complexity: O(1) - Constant space used for tracking the answer.

class Solution {
public:
    int uniqueXorTriplets(vector<int>& nums) {
        int n = nums.size();

        // Base cases for n = 1 and n = 2
        if(n == 1 || n == 2)
            return n;
        
        int ans = 1; // Start with 2^0
        
        // Find the nearest power of 2 strictly greater than n
        while(ans <= n) {
            ans = (ans << 1); // Equivalent to ans *= 2
        }

        return ans;
    }
};

/*
*
* Dry Run
* 1. nums = [3, 1, 2, 4, 5], n = 5
* 2. n is not 1 or 2, continue
* 3. ans = 1
* 4. ans (1) <= n (5): ans = 1 << 1 (ans becomes 2)
* 5. ans (2) <= n (5): ans = 2 << 1 (ans becomes 4)
* 6. ans (4) <= n (5): ans = 4 << 1 (ans becomes 8)
* 7. ans (8) <= n (5) is False. Exit loop.
* 8. Return ans = 8
*
*/
```

## 💡 Key Insights
* **Pattern Recognition:** Tracing constraints and results for incremental values of $N$ ($1, 2, 3, 4...$) reveals a distinct pattern tying back to the binary representation lengths.
* **XOR Limitations:** Bitwise XOR cannot exceed the maximum bit limit defined by the highest number being XOR'd. If the maximum element is represented in 3 bits, the maximum possible XOR output also requires at most 3 bits.
* **Trivial Permutations:** The array being a "permutation" is a distractor. The math is dependent entirely on the value $N$.
