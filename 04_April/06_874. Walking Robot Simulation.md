## 📜 Problem Statement

A robot on an infinite XY-plane starts at point `(0, 0)` facing **north**. The robot receives an array of integers `commands`, which represents a sequence of moves that it needs to execute. There are only three possible types of instructions the robot can receive:

* `-2`: Turn left 90 degrees.
* `-1`: Turn right 90 degrees.
* `1 <= k <= 9`: Move forward `k` units, one unit at a time.

Some grid squares contain **obstacles**. The $i^{th}$ obstacle is at grid point `obstacles[i] = (xi, yi)`. If the robot runs into an obstacle, it will stay in its current location (on the block adjacent to the obstacle) and move onto the next command.

**Return** the *maximum squared Euclidean distance* that the robot reaches at any point in its path (i.e. if the distance is $5$, return $25$).

> **Note:**
> * There can be an obstacle at `(0, 0)`. If this happens, the robot will ignore the obstacle until it has moved off the origin. However, it will be unable to return to `(0, 0)` due to the obstacle.
> * North means +Y direction.
> * East means +X direction.
> * South means -Y direction.
> * West means -X direction.

---

## ⚙️ Constraints

* $1 \le commands.length \le 10^4$
* `commands[i]` is either `-2`, `-1`, or an integer in the range `[1, 9]`.
* $0 \le obstacles.length \le 10^4$
* $-3 \times 10^4 \le x_i, y_i \le 3 \times 10^4$
* The answer is guaranteed to be less than $2^{31}$.

## 🧠 Approach & Explanation

This is a clear **simulation** problem. We need to follow the rules given to us step-by-step. 

1. **Direction Representation:** We can map the directions cleanly using a coordinate pair. Starting at North, the movement adds `(0, 1)` to our current coordinate. 
   When changing direction, we can use simple coordinate geometry:
   * **Turn Left 90° (-2):** The current direction `(x, y)` becomes `(-y, x)`.
   * **Turn Right 90° (-1):** The current direction `(x, y)` becomes `(y, -x)`.
   
2. **Handling Obstacles Efficiently:**
   Since we move unit by unit, we must verify if the very next coordinate holds an obstacle. We are given the obstacles as a 2D array, which means searching for a coordinate would take $O(m)$ time per step. To optimize this to $O(1)$, we hash the obstacle coordinates into an `unordered_set`. 
   
   *String Hashing Trick:* To avoid collisions like `(11, 0)` converting to `"110"` and `(1, 10)` also converting to `"110"`, we must add a delimiter. We construct keys as `"x_y"` to uniquely identify obstacles.

3. **Simulating the Moves:**
   * Read the command.
   * If it's `-1` or `-2`, update the direction vector.
   * If it's a positive number $K$, take $K$ individual $1$-unit steps. Before committing to a step, construct the string key for the target cell and verify its presence in the set. If blocked, abort the remaining steps for that command.
   * Calculate and update the maximum distance squared at every valid coordinate reached.

## 🚀 Solution Code

```cpp
// Intuition: The problem requires simulating a robot's movement on a grid with obstacles. Since obstacles block the path, we need to move step-by-step and check for obstacles at each step. To quickly check if a coordinate is an obstacle, a HashSet of strings ("x_y") is optimal.
// Approach:
// 1. Store all obstacle coordinates as string keys ("x_y") in an unordered_set for O(1) lookups.
// 2. Initialize the robot's position at (0, 0) and direction facing North (0, 1).
// 3. Iterate through the commands. If the command is a turn (-1 or -2), update the direction vector using 90-degree rotation formulas.
// 4. If the command is a move, step forward one unit at a time up to 'k' steps.
// 5. At each step, calculate the next coordinate and check if it's in the obstacle set. If it is, break out of the moving loop early.
// 6. Update the maximum distance squared after each command.
// Time Complexity: O(M + N * K), where M is the number of obstacles, N is the number of commands, and K is the maximum steps per command (up to 9).
// Space Complexity: O(M) for storing the obstacles in the unordered_set.

class Solution {
public:
    int robotSim(vector<int>& commands, vector<vector<int>>& obstacles) {
        unordered_set<string> st;
        
        // Insert obstacles into the set using a string delimiter to avoid collisions
        for(vector<int>& obs : obstacles) {
            string key = to_string(obs[0]) + "_" + to_string(obs[1]);
            st.insert(key);
        }

        int x = 0;
        int y = 0;
        int maxD = 0;

        // Pointing to North initially
        pair<int, int> dir = {0, 1}; 

        // Process each command
        for(int i = 0; i < commands.size(); i++) {
            if(commands[i] == -2) { 
                // Turn left 90 degrees: (x, y) -> (-y, x)
                dir = {-dir.second, dir.first};
            } else if(commands[i] == -1) { 
                // Turn right 90 degrees: (x, y) -> (y, -x)
                dir = {dir.second, -dir.first};
            } else { 
                // Move forward step-by-step
                for(int step = 0; step < commands[i]; step++) {
                    int newX = x + dir.first;
                    int newY = y + dir.second;

                    // Create the key for the next coordinate
                    string nextKey = to_string(newX) + "_" + to_string(newY);

                    // Stop moving if an obstacle is encountered
                    if(st.find(nextKey) != st.end()) {
                        break;
                    }

                    x = newX;
                    y = newY;
                }
            }

            // Keep track of the maximum Euclidean distance squared
            maxD = max(maxD, x*x + y*y);
        }

        return maxD;
    }
};

/*
 *
 * Dry Run
 *
 * Input: commands = [4,-1,3], obstacles = []
 * Initialization:
 * x = 0, y = 0, maxD = 0, dir = (0, 1) [North]
 * * Command 4:
 * - Move forward 4 steps:
 * step 1: newX = 0, newY = 1
 * step 2: newX = 0, newY = 2
 * step 3: newX = 0, newY = 3
 * step 4: newX = 0, newY = 4
 * - Update x = 0, y = 4
 * - maxD = max(0, 0^2 + 4^2) = 16
 * * Command -1:
 * - Turn right 90 degrees
 * - dir becomes (dir.second, -dir.first) -> (1, 0) [East]
 * * Command 3:
 * - Move forward 3 steps:
 * step 1: newX = 1, newY = 4
 * step 2: newX = 2, newY = 4
 * step 3: newX = 3, newY = 4
 * - Update x = 3, y = 4
 * - maxD = max(16, 3^2 + 4^2) = 25
 * * Result: 25
 *
 */
````

## 🔍 Key Insights

1.  **Delimiter Injection Strategy:** When converting an array/matrix coordinates into a unified string key, always use delimiters (e.g., `_`, `#`, `,`). Failing to do so combines coordinates like `(1, 10)` and `(11, 0)` into the exact same string `"110"`, creating catastrophic false positives.
2.  **Matrix Rotations for Orientation:** Knowing that a vector `(dx, dy)` becomes `(-dy, dx)` when rotated $90^\circ$ counter-clockwise (Left) and `(dy, -dx)` when rotated clockwise (Right) eliminates the need to map strings like "North", "South", "East", "West" and heavily simplifies transition code.
3.  **Step-by-Step Traversal:** Never teleport the robot mathematically (`x += dx * K`). Instead, simulate atomic unit-by-unit movements because an obstacle might interrupt the movement path immediately.
