## 📖 Problem Statement

There are `n` **1-indexed** robots, each having a position on a line, health, and movement direction.

You are given **0-indexed** integer arrays `positions`, `healths`, and a string `directions` (`directions[i]` is either **'L'** for left or **'R'** for right). All integers in `positions` are **unique**.

All robots start moving on the line **simultaneously** at the **same speed** in their given directions. If two robots ever share the same position while moving, they will **collide**.

If two robots collide, the robot with **lower health** is **removed** from the line, and the health of the other robot **decreases** by one. The surviving robot continues in the **same** direction it was going. If both robots have the **same** health, they are both **removed from the line**.

Your task is to determine the **health** of the robots that survive the collisions, in the same **order** that the robots were given, i.e., final health of robot 1 (if survived), final health of robot 2 (if survived), and so on. If there are no survivors, return an empty array.

Return an array containing the health of the remaining robots (in the order they were given in the input), after no further collisions can occur.

> **Note:** The `positions` array may be unsorted.

---

## 🎯 Constraints

- `1 <= positions.length == healths.length == directions.length == n <= 10^5`
- `1 <= positions[i], healths[i] <= 10^9`
- `directions[i] == 'L'` or `directions[i] == 'R'`
- All values in `positions` are distinct.

## 🧠 Approach

This problem is closely related to the classic "Asteroid Collision" problem, and a **Stack** is the ideal data structure to simulate these mechanics.

1. **Sequential Processing:** Because the given positions are unsorted, we must process the robots as they appear from left to right on the 1D number line. To achieve this without losing track of their original input indices (which we need for modifying their healths and returning the final array in order), we create a separate array called `indices` filled with `[0, 1, 2, ..., n-1]`.
2. **Sorting by Position:** We sort this `indices` array based on the actual values inside the `positions` array. This gives us an iteration order corresponding exactly to the left-to-right layout on the number line.
3. **Simulating with a Stack:** - We iterate through the sorted `indices`.
   - If a robot is moving Right (`'R'`), we push its original index onto the stack. These robots are waiting for a potential collision with any robot coming from the right.
   - If a robot is moving Left (`'L'`), it will collide with the robots currently at the top of the stack (which are moving `'R'`).
4. **Resolving Collisions:** While the stack is not empty and the current left-moving robot still has health > 0:
   - Compare its health with the top of the stack.
   - If the stack's top robot is stronger, it destroys the left-moving robot, and the stack's top robot loses 1 health.
   - If the left-moving robot is stronger, it destroys the stack's top robot (which is popped), loses 1 health, and continues colliding with the next robot down the stack.
   - If they are equal, both are destroyed.
5. **Final Output:** The easiest way to format the result is to simply iterate through the original `healths` array from `0` to `n-1`. Since we modified the healths in place (setting destroyed robots to `0`), any robot with health `> 0` survived. Appending them in this loop naturally preserves the required original input order!

## 💻 Solution Code

```cpp
// Intuition: The problem simulates collisions on a 1D line, making it a perfect fit for a Stack (similar to Asteroid Collision). We process robots from left to right on the number line.
// Approach:
// 1. Create an array of actual indices (0 to n-1).
// 2. Sort these indices based on the robots' positions to process them sequentially from left to right.
// 3. Use a stack to track indices of robots moving to the Right ('R').
// 4. Iterate through the sorted indices. If moving 'R', push to stack. If moving 'L', resolve collisions with the stack's top robots based on their healths until the 'L' robot is destroyed or the stack is empty.
// 5. Finally, iterate through the original healths array and collect any non-zero healths.
// Time Complexity: O(n log n) primarily due to sorting the indices. The stack operations take O(n) overall as each element is pushed/popped at most once.
// Space Complexity: O(n) for storing the sorted indices array and the stack.

//Using Stack
//T.C : O(nlogn)
//T.C : O(n)
class Solution {
public:
    vector<int> survivedRobotsHealths(vector<int>& positions, vector<int>& healths, string directions) {
        int n = positions.size();
        vector<int> indices(n);
        
        iota(indices.begin(), indices.end(), 0); //This will fill the array as -> 0, 1, 2, 3, 4, n-1
        stack<int> st;

        auto lambda = [&](int i, int j) {
            return positions[i] < positions[j];
        };

        sort(begin(indices), end(indices), lambda);

        vector<int> result;
        for (int currentIndex : indices) {
            if (directions[currentIndex] == 'R') {
                st.push(currentIndex); // Right-moving robot waits for collision
            } else {
                while (!st.empty() && healths[currentIndex] > 0) {
                    int topIndex = st.top();
                    st.pop();

                    if (healths[topIndex] > healths[currentIndex]) {
                        healths[topIndex] -= 1; // Top survives, health decreases
                        healths[currentIndex] = 0; // Current robot destroyed
                        st.push(topIndex);
                    } else if (healths[topIndex] < healths[currentIndex]) {
                        healths[currentIndex] -= 1; // Current survives, health decreases
                        healths[topIndex] = 0; // Top robot destroyed
                    } else {
                        healths[currentIndex] = 0; // Both destroyed
                        healths[topIndex] = 0;
                    }
                }
            }
        }

        for (int i = 0; i < n; ++i) {
            if (healths[i] > 0) {
                result.push_back(healths[i]); // Append surviving robots in original order
            }
        }
        return result;
    }
};

/*
*
* Dry Run
*
* Input: positions = [3,5,2,6], healths = [10,10,15,12], directions = "RLRL"
* n = 4. Original indices = [0, 1, 2, 3]
* Sorted indices based on positions ([2, 3, 5, 6]): [2, 0, 1, 3]
*
* Step 1: currentIndex = 2 (Position 2, Health 15, Dir 'R')
* directions[2] == 'R'. Push 2 to stack. st = [2]
* * Step 2: currentIndex = 0 (Position 3, Health 10, Dir 'R')
* directions[0] == 'R'. Push 0 to stack. st = [2, 0]
*
* Step 3: currentIndex = 1 (Position 5, Health 10, Dir 'L')
* directions[1] == 'L'. Collision with st.top() = 0
* healths[0] (10) == healths[1] (10). Both destroyed.
* healths[0] = 0, healths[1] = 0. st.pop() -> st = [2]
* Loop ends for currentIndex = 1 since its health is 0.
*
* Step 4: currentIndex = 3 (Position 6, Health 12, Dir 'L')
* directions[3] == 'L'. Collision with st.top() = 2
* healths[2] (15) > healths[3] (12). Right-moving survives.
* healths[2] becomes 14. healths[3] becomes 0.
* st.push(2). st = [2].
* Loop ends for currentIndex = 3.
*
* Final healths array: [0, 0, 14, 0]
* Iterating original order: result = [14]
*
*/
```

## 🔑 Key Insights

- **Index Tracking is Crucial:** The most critical trick in this problem isn't the stack itself, but creating the `indices` array. By sorting `indices` based on `positions`, we decouple the *spatial order* of the robots from their *output order*. 
- **In-Place Health Modification:** Using the original `healths` array to store `0` for dead robots eliminates the need to map surviving robots back to their original positions. A simple linear scan over `healths` at the very end automatically yields the correct results in the correct order.
- **Amortized Stack Time:** The inner `while` loop pops elements from the stack. Even though there is a loop inside a loop, an element is pushed and popped at most once, making the stack operations perfectly `O(n)`. The bottleneck of the algorithm is strictly the sorting step `O(n log n)`.
