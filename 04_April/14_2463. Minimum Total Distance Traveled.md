## 📝 Problem Statement

There are some robots and factories on the X-axis. You are given an integer array `robot` where `robot[i]` is the position of the `i-th` robot. You are also given a 2D integer array `factory` where `factory[j] = [position_j, limit_j]` indicates that `position_j` is the position of the `j-th` factory and that the `j-th` factory can repair at most `limit_j` robots.

The positions of each robot are **unique**. The positions of each factory are also **unique**. Note that a robot can be **in the same position** as a factory initially.

All the robots are initially broken; they keep moving in one direction. The direction could be the negative or the positive direction of the X-axis. When a robot reaches a factory that did not reach its limit, the factory repairs the robot, and it stops moving.

At any moment, you can set the initial direction of moving for **some** robot. Your target is to minimize the total distance traveled by all the robots.

Return *the minimum total distance traveled by all the robots*. The test cases are generated such that all the robots can be repaired.

**Notes:**
* All robots move at the same speed.
* If two robots move in the same direction, they will never collide.
* If two robots move in opposite directions and they meet at some point, they do not collide. They cross each other.
* If a robot passes by a factory that reached its limits, it crosses it as if it does not exist.
* If the robot moved from a position `x` to a position `y`, the distance it moved is `|y - x|`.

- `1 <= robot.length, factory.length <= 100`
- `factory[j].length == 2`
- `-10^9 <= robot[i], position_j <= 10^9`
- `0 <= limit_j <= robot.length`
- The input will be generated such that it is always possible to repair every robot.

## 🧠 Approach & Explanation

To minimize the total distance, we intuitively want the closest robots to be paired with the closest factories. However, dynamically managing the capacities of each factory while finding the optimal matching can become highly complex.

Here is the step-by-step strategy to elegantly solve this:

1. **Sort Positions (The Spatial Alignment):**
   Sort both the `robot` array and the `factory` array based on their positions on the X-axis. Sorting ensures that as we iterate through the robots from left to right, we can match them linearly with the factories from left to right, preventing paths from overlapping unnecessarily.

2. **Flatten the Factory Array (Capacity Simplification):**
   A factory at position `P` with a limit `L` means it can fix `L` robots. Instead of maintaining a complex state for the remaining limit of each factory, we can "flatten" or "expand" the factory array. 
   If a factory is at position `2` with limit `2`, we treat it as two separate, identical factory slots: `[2, 2]`. This way, every single element in our new `positions` array represents **exactly one available repair slot**.

3. **Dynamic Programming (Take or Skip):**
   Now the problem reduces to matching `M` robots to `N` available factory slots linearly. For the current robot `ri` and the current factory slot `fi`, we have two choices:
   - **Take:** The current robot uses the current factory slot. The cost is `abs(robot[ri] - positions[fi])`. We then proceed to match the *next* robot (`ri + 1`) with the *next* factory slot (`fi + 1`).
   - **Skip:** The current robot decides NOT to use the current factory slot (reserving it for a future robot, or because a better factory lies ahead). We leave the robot at `ri` and move to explore the *next* factory slot `fi + 1`.

4. **Memoization:**
   Because there are many overlapping subproblems (e.g., reaching the same robot and factory slot index through different combinations of skips), we use a 2D Dynamic Programming table `t[ri][fi]` to cache the minimum distance calculated for that state.

## 💻 Solution Code

```cpp
// Intuition: Sorting aligns robots and factories spatially. Expanding factories simplifies limit tracking. The problem then reduces to finding the optimal sequence of matching a robot to a factory slot using a "Take or Skip" dynamic programming approach.
// Approach:
// 1. Sort both the robots and the factories arrays by position.
// 2. Expand the factory array into a 1D `positions` array where each factory position is repeated `limit` times.
// 3. Use recursion with memoization: For each robot and factory position, try two choices:
//    a) Assign the current robot to the current factory position.
//    b) Skip the current factory position and try the next one for the current robot.
// 4. Return the minimum distance among the valid choices.
// Time Complexity: O(m * n), where m is the number of robots and n is the total number of expanded factory positions (sum of all limits). Each state (ri, fi) is computed at most once.
// Space Complexity: O(m * n) to store the DP table, plus O(m + n) for the recursion stack.

class Solution {
public:
    typedef long long ll;

    ll solve(int ri, int fi, vector<int>& robot, vector<int>& positions, vector<vector<ll>>& t) {
        // Base Case 1: All robots successfully repaired
        if(ri >= robot.size()) {
            return 0; // no more distance to be covered
        }

        // Base Case 2: Factory slots exhausted but robots remain (Invalid path)
        if(fi >= positions.size()) {
            return 1e12; // Use a large valid bound instead of INT_MAX to avoid overflow
        }

        // Return memoized result if available
        if(t[ri][fi] != -1) {
            return t[ri][fi];
        }

        // Option 1: Take the current factory slot for the current robot
        ll take_current_factory = abs(robot[ri] - positions[fi]) + solve(ri+1, fi+1, robot, positions, t);
        
        // Option 2: Skip the current factory slot, try the next slot for the SAME robot
        ll skip = solve(ri, fi+1, robot, positions, t);

        // Store and return the minimum of taking or skipping
        return t[ri][fi] = min(take_current_factory, skip);
    }

    long long minimumTotalDistance(vector<int>& robot, vector<vector<int>>& factory) {
        // Step 1: Sort spatially for linear optimal matching
        sort(begin(robot), end(robot));
        sort(begin(factory), end(factory));

        int m = robot.size();
        
        // Step 2: Expand factory positions to avoid tracking capacities dynamically
        vector<int> positions;
        for(int i = 0; i < factory.size(); i++) {
            int limit = factory[i][1];
            int pos   = factory[i][0];

            // Add the position 'limit' number of times
            for(int j = 0; j < limit; j++) {
                positions.push_back(pos);
            }
        }
        
        int n = positions.size();
        
        // DP table initialization with -1
        vector<vector<ll>> t(m+1, vector<ll>(n+1, -1));

        // Step 3: Trigger the recursive solver from the 0th robot and 0th factory slot
        return solve(0, 0, robot, positions, t);
    }
};

/*
*
* Dry Run
*
* Input: robot = [0,4,6], factory = [[2,2],[6,2]]
* * 1. Sorting:
* robot = [0, 4, 6]
* factory = [[2,2], [6,2]]
* * 2. Flattening Factories:
* positions = [2, 2, 6, 6]
* * 3. Recursive Execution (solve(ri, fi)):
* solve(0, 0) -> robot[0]=0, pos[0]=2
* Option Take: abs(0-2) + solve(1, 1) = 2 + solve(1, 1)
* Option Skip: solve(0, 1)
*
* Evaluating solve(1, 1) -> robot[1]=4, pos[1]=2
* Option Take: abs(4-2) + solve(2, 2) = 2 + solve(2, 2)
* Option Skip: solve(1, 2)
* * Evaluating solve(2, 2) -> robot[2]=6, pos[2]=6
* Option Take: abs(6-6) + solve(3, 3) = 0 + 0 = 0 (Base case ri=3 returns 0)
* Option Skip: solve(2, 3)
* * Rolling back:
* solve(2, 2) returns 0.
* solve(1, 1) calculates Take = 2 + 0 = 2.
* solve(0, 0) calculates Take = 2 + 2 = 4.
* * Output: 4
*
*/
```

## 🔍 Key Insights

- **State Expansion:** The biggest challenge in DP problems with limits/capacities is passing the remaining capacity as state. Expanding the array completely neutralizes this complication, translating the problem into a simple 1-to-1 matching problem.
- **Overflow Handling:** When a state is invalid (no factories left for remaining robots), we return `1e12` instead of `INT_MAX`. Since we add distance to the recursive result, returning `INT_MAX` would cause integer overflow errors.
- **Why Sorting Works:** Sorting both inputs prevents crossing paths. In an optimal assignment, if robot `A` is to the left of robot `B`, the factory assigned to `A` will never be to the right of the factory assigned to `B`.
