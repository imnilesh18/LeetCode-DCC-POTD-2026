## 📝 Problem Statement

A `width x height` grid is on an XY-plane with the **bottom-left** cell at `(0, 0)` and the **top-right** cell at `(width - 1, height - 1)`. The grid is aligned with the four cardinal directions ("North", "East", "South", and "West"). A robot is **initially** at cell `(0, 0)` facing direction `"East"`.

The robot can be instructed to move for a specific number of **steps**. For each step, it does the following:
1. Attempts to move **forward one** cell in the direction it is facing.
2. If the cell the robot is **moving to** is **out of bounds**, the robot instead **turns 90 degrees counterclockwise** and retries the step.

After the robot finishes moving the number of steps required, it stops and awaits the next instruction.

Implement the `Robot` class:
- `Robot(int width, int height)` Initializes the `width x height` grid with the robot at `(0, 0)` facing `"East"`.
- `void step(int num)` Instructs the robot to move forward `num` steps.
- `int[] getPos()` Returns the current cell the robot is at, as an array of length 2, `[x, y]`.
- `String getDir()` Returns the current direction of the robot, `"North"`, `"East"`, `"South"`, or `"West"`.

---

## 🚫 Constraints

- `2 <= width, height <= 100`
- `1 <= num <= 10^5`
- At most `10^4` calls **in total** will be made to `step`, `getPos`, and `getDir`.

## 🧠 Approach & Explanation

### The Core Insight
The most critical observation for this problem is that the robot **only moves along the perimeter (boundary)** of the grid. It never goes into the inner cells. Because of this, the robot travels in a fixed, cyclic path endlessly. 

Instead of moving the robot step-by-step (which would cause a Time Limit Exceeded error given `num` can be up to `10^5`), we can **precompute** the entire boundary.

### Step-by-Step Strategy
1. **Precomputing the Boundary:** We can trace the boundary cells once during initialization—going East along the bottom, North up the right side, West across the top, and South down the left side. We store the `(x, y)` coordinates and the direction the robot will face at each cell in an array called `pos`.
2. **O(1) Step Updates:** With the boundary precomputed, simulating `num` steps becomes a simple modulo arithmetic operation. The new index is just `(currentIndex + num) % totalBoundaryCells`.
3. **Handling the (0, 0) Edge Case:** - Before any movement, the robot at `(0, 0)` faces **East**.
   - If the robot completes a full lap around the boundary and lands exactly back at `(0, 0)`, it will be coming down the left edge, thus facing **South**.
   - To resolve this elegantly, we modify the precomputed direction at index `0` to be South. We then introduce a boolean flag `moved`. If `getDir()` is called and `moved` is still false, we hardcode the return to `"East"`. Otherwise, we just return the precomputed direction.

## 💻 Optimized Solution (C++)

```cpp
// Intuition: The robot only moves along the boundary/perimeter of the grid. Precomputing the boundary cells allows O(1) position lookups using modulo arithmetic for the steps.
// Approach:
// 1. Store the entire perimeter's coordinates (x, y) and direction (0=East, 1=North, 2=West, 3=South) in an array.
// 2. Fill the array by traversing the bottom row, right column, top row, and left column.
// 3. To handle the corner case where the robot loops back to (0,0), change the stored direction at index 0 to South.
// 4. Use a `moved` boolean flag. If queried before moving, it faces East. Otherwise, use the precomputed direction.
// 5. Update index for `step(num)` as `(idx + num) % perimeter_size`.
// Time Complexity: O(width + height) for Constructor, O(1) for step, getPos, and getDir.
// Space Complexity: O(width + height) to store the perimeter cells.

class Robot {
public:
    int idx = 0;
    bool moved = false;
    vector<vector<int>> pos; // Stores all perimeter/boundary cell positions

    Robot(int width, int height) {
        // Bottom Row (left to right) -> Facing East (0)
        for(int x = 0; x < width; x++) {
            pos.push_back({x, 0, 0});
        }

        // Right Col (bottom to top) -> Facing North (1)
        for(int y = 1; y < height; y++) {
            pos.push_back({width - 1, y, 1});
        }

        // Top row (right to left) -> Facing West (2)
        for(int x = width - 2; x >= 0; x--) {
            pos.push_back({x, height - 1, 2});
        }

        // Left column (top to bottom) -> Facing South (3)
        for(int y = height - 2; y > 0; y--) {
            pos.push_back({0, y, 3});
        }

        // Make it South for handling corner case when it comes to (0, 0) after a full cycle
        pos[0][2] = 3; 
    }
    
    void step(int num) {
        moved = true; // Mark as moved
        // O(1) state update using modulo arithmetic
        idx = (idx + num) % pos.size();
    }
    
    vector<int> getPos() {
        // Return precomputed x and y
        return {pos[idx][0], pos[idx][1]};
    }
    
    string getDir() {
        // If it never moved, it still faces East
        if(!moved) return "East";

        int d = pos[idx][2];

        // Map integer back to string direction
        if(d == 0) return "East";
        else if(d == 1) return "North"; 
        else if(d == 2) return "West"; 
        return "South";
    }
};

/*
*
* Dry Run
*
* Robot robot = new Robot(6, 3);
* Precomputes boundary size: 6 + 2 + 5 + 1 = 14 cells.
* pos[0] = {0, 0, 3} (South - overridden for full cycle)
* Initially, idx = 0, moved = false.
* * robot.step(2);
* moved = true
* idx = (0 + 2) % 14 = 2.
* pos[2] = {2, 0, 0} (East)
*
* robot.step(2);
* idx = (2 + 2) % 14 = 4.
* pos[4] = {4, 0, 0} (East)
*
* robot.getPos(); -> returns [4, 0]
* robot.getDir(); -> pos[4][2] is 0 -> returns "East"
* * robot.step(2);
* idx = (4 + 2) % 14 = 6.
* pos[6] = {5, 1, 1} (North)
* * robot.step(1);
* idx = (6 + 1) % 14 = 7.
* pos[7] = {5, 2, 1} (North)
*
* robot.step(4);
* idx = (7 + 4) % 14 = 11.
* pos[11] = {1, 2, 2} (West)
*
* robot.getPos(); -> returns [1, 2]
* robot.getDir(); -> pos[11][2] is 2 -> returns "West"
*
*/
```
