## 📖 Problem Statement

You are given a **0-indexed** integer array `nums`. There exists an array `arr` of length `nums.length`, where `arr[i]` is the sum of `|i - j|` over all `j` such that `nums[j] == nums[i]` and `j != i`. If there is no such `j`, set `arr[i]` to be `0`.

Return the array `arr`.

> **Note:** This question is the same as **2121: Intervals Between Identical Elements**.

---

## 🚫 Constraints

- `1 <= nums.length <= 10^5`
- `0 <= nums[i] <= 10^9`

## 🧠 Approach & Explanation

### The Naive Intuition (Brute Force)
A straightforward brute force approach requires evaluating every index `i`, running a secondary pointer `j` through the entire array, and calculating `|i - j|` whenever `nums[i] == nums[j]`. 
While this is logical, taking a number `N` up to $10^5$, an $O(N^2)$ algorithm will confidently result in a **Time Limit Exceeded (TLE)** error. However, closely observing the repetitive actions in the brute-force method points directly to the optimized solution.

### The Optimized Mathematical Breakdown
When standing at an index `i`, identical elements are scattered on both the **Left Hand Side (LHS)** and the **Right Hand Side (RHS)**. 

Let's dissect the distance calculation:
1. **For Left Hand Side ($j < i$):**
   The absolute distance formula `|i - j|` simply becomes `i - j`.
   Summing this over multiple occurrences on the left gives:
   `(i - j1) + (i - j2) + (i - j3) ...`
   This equals: `(Count of occurrences on left * i) - (Sum of their indices)`.

2. **For Right Hand Side ($j > i$):**
   The absolute distance formula `|i - j|` evaluates to `j - i`.
   Summing this over multiple occurrences on the right gives:
   `(j1 - i) + (j2 - i) + (j3 - i) ...`
   This equals: `(Sum of their indices) - (Count of occurrences on right * i)`.

By leveraging **Prefix Sums** and **Hash Maps**, we can track two vital pieces of information dynamically as we traverse:
1. **Frequency:** How many times we've seen a specific number so far.
2. **Index Sum:** The sum of indices where we previously saw this specific number.

**Execution Steps:**
1. Traverse left-to-right to compute and store the LHS distance contributions dynamically using maps.
2. Clear the maps to reuse them.
3. Traverse right-to-left to compute and add the RHS distance contributions to our result array.
4. Return the fully computed result array.

## 💻 Solution Code

### 🐌 Brute-force Solution (C++)
```cpp
class Solution {
public:
    vector<long long> distance(vector<int>& nums) {
        int n = nums.size();
        vector<long long> arr(n, 0);
        
        for (int i = 0; i < n; i++) {
            long long sum = 0;
            for (int j = 0; j < n; j++) {
                if (i != j && nums[i] == nums[j]) {
                    sum += abs(i - j);
                }
            }
            arr[i] = sum;
        }
        return arr;
    }
};
```

### 🚀 Optimized Solution (C++)
```cpp
// Intuition:
// To avoid the O(N^2) brute force, calculate distance contributions from identical elements on the left and right independently.
// Left contribution for index `i`: (frequency of num on left * i) - (sum of their indices).
// Right contribution for index `i`: (sum of their indices on right) - (frequency of num on right * i).

// Approach:
// 1. Initialize two unordered maps to track the sum of indices and the frequency of each element.
// 2. Iterate left to right: calculate left-side distances, update result array, and update maps.
// 3. Clear maps to prepare for the right-to-left pass.
// 4. Iterate right to left: calculate right-side distances, add to the existing result array, and update maps.
// 5. Return the finalized distance array.

// Time Complexity: O(n) - We traverse the array exactly twice. Unordered map lookups and insertions average O(1) time.
// Space Complexity: O(n) - The maps will store at most N unique elements, resulting in O(N) space.

class Solution {
public:
    typedef long long ll;

    vector<ll> distance(vector<int>& nums) {
        int n = nums.size();

        vector<ll> arr(n, 0); // stores the final answer

        unordered_map<int, ll> indexSum;   // nums[i] -> sum of indices
        unordered_map<int, ll> indexCount; // nums[i] -> frequency

        // Left to Right Pass
        for(int i = 0; i < n; i++) {
            ll freq = indexCount[nums[i]]; // frequency on left
            ll sum  = indexSum[nums[i]];   // sum of indices on left

            arr[i] += freq * i - sum;      // calculate and add left contribution

            indexCount[nums[i]] += 1;      // update frequency
            indexSum[nums[i]]   += i;      // update sum of indices
        }

        indexSum.clear();   // clear maps for the next pass
        indexCount.clear();

        // Right to Left Pass
        for(int i = n - 1; i >= 0; i--) {
            ll freq = indexCount[nums[i]]; // frequency on right
            ll sum  = indexSum[nums[i]];   // sum of indices on right

            arr[i] += sum - freq * i;      // calculate and add right contribution

            indexCount[nums[i]] += 1;      // update frequency
            indexSum[nums[i]]   += i;      // update sum of indices
        }

        return arr;
    }
};

/*

* Dry Run
*

*
* --- Left to Right Pass ---
* i = 0, nums[0] = 1. freq = 0, sum = 0. arr[0] += 0*0 - 0 = 0. Map updates: 1->count:1, sum:0
* i = 1, nums[1] = 3. freq = 0, sum = 0. arr[1] += 0*1 - 0 = 0. Map updates: 3->count:1, sum:1
* i = 2, nums[2] = 1. freq = 1, sum = 0. arr[2] += 1*2 - 0 = 2. Map updates: 1->count:2, sum:2
```

## 📌 Notes & Key Insights
- **Key Insight:** Avoid nested loops by algebraically refactoring repetitive subtraction formulas. Breaking distances strictly into left-sided relationships and right-sided relationships shifts the complexity from geometric combinations to simple arithmetic scaling `O(1)` per index.
- **Data Types:** Utilizing `long long` (a 64-bit integer) is mandatory. Index summations and multiplied scales for an array sized $10^5$ will quickly overflow standard 32-bit `int` limits.
- **Modular Map Re-usage:** `clear()` operations efficiently recycle memory limits, allowing for $O(N)$ auxiliary space rather than risking duplicate structures.
