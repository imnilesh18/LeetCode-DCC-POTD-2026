## 📝 Problem Statement

You are given an array of positive integers `arr`. Perform some operations (possibly none) on `arr` so that it satisfies these conditions:
1. The value of the first element in `arr` must be `1`.
2. The absolute difference between any 2 adjacent elements must be less than or equal to `1`. In other words, `abs(arr[i] - arr[i - 1]) <= 1` for each `i` where `1 <= i < arr.length` (0-indexed). `abs(x)` is the absolute value of `x`.

There are 2 types of operations that you can perform any number of times:
*   **Decrease** the value of any element of `arr` to a smaller positive integer.
*   **Rearrange** the elements of `arr` to be in any order.

Return the **maximum possible value** of an element in `arr` after performing the operations to satisfy the conditions.

---

## 🚦 Constraints

*   `1 <= arr.length <= 10^5`
*   `1 <= arr[i] <= 10^9`

---

## 💡 Examples

| Example | Input | Output | Explanation |
| :--- | :--- | :--- | :--- |
| **1** | `arr = [2,2,1,2,1]` | `2` | Rearrange `arr` to `[1,2,2,2,1]`. The largest element is `2`. |
| **2** | `arr = [100,1,1000]` | `3` | Rearrange to `[1,100,1000]`. Decrease `100` to `2` and `1000` to `3`. The array becomes `[1,2,3]`. The largest element is `3`. |
| **3** | `arr = [1,2,3,4,5]` | `5` | The array already satisfies the conditions, and the largest element is `5`. |

## 🔍 Key Insights

*   **"Rearranging" implies Sorting:** The problem allows us to rearrange the elements in any order. To minimize the difference between adjacent elements, the optimal strategy is to sort the array first.
*   **Greedy Choice:** We want to maximize the elements, but we are restricted by the `abs(arr[i] - arr[i-1]) <= 1` rule. If an element is too large, we should decrease it to exactly `arr[i-1] + 1` to keep it as large as legally possible.
*   **One-Way Operation:** We can only *decrease* elements. We cannot increase them. Therefore, if `arr[i]` is already smaller than or equal to `arr[i-1] + 1`, we simply leave it as is.

---

## 🛠️ Approach

1.  **Sort the Array:** Sorting guarantees that smaller numbers are processed first, which perfectly aligns with our goal of building a sequence that increments by at most 1.
2.  **Enforce Condition 1:** The first element must be `1`. So, we explicitly set `arr[0] = 1`.
3.  **Iterate and Enforce Condition 2 (Greedy):** Loop through the array starting from index 1.
    *   Calculate the absolute difference between `arr[i]` and `arr[i-1]`.
    *   If the difference is greater than `1`, we must reduce `arr[i]`. To maximize our final answer, we reduce `arr[i]` to exactly `arr[i-1] + 1`.
4.  **Track the Maximum:** Keep a running maximum of the elements after they have been processed. Since the array is sorted and we only step up by at most 1, the last element will naturally be the maximum.

## 💻 Solution Code

### Brute-force Solution (Conceptual)
A true brute-force solution would involve generating all permutations of the array and trying to reduce elements to find valid sequences, tracking the maximum element found across all valid configurations. This would take `O(N!)` time and result in a Time Limit Exceeded (TLE) error, making it entirely impractical for `N = 10^5`.

```cpp
// Intuition: The ability to "rearrange" suggests sorting. By sorting, we can greedily enforce the rules from left to right, reducing elements only when necessary to maintain a maximum step of 1.
// Approach: Sort the array, force the first element to 1, and then iterate through the rest. If the difference between adjacent elements is > 1, greedily reduce the current element to the previous element + 1. Keep track of the maximum value seen.
// Time Complexity: O(N log N) where N is the size of the array, due to the sorting step. The subsequent linear scan takes O(N) time.
// Space Complexity: O(1) or O(log N) depending on the sorting algorithm used under the hood by C++ std::sort. We modify the array in-place.

class Solution {
public:
    int maximumElementAfterDecrementingAndRearranging(vector<int>& arr) {
        int n = arr.size();

        // Sort the array to process smaller elements first
        sort(begin(arr), end(arr));

        int maxEl = -1;

        for(int i = 0; i < n; i++) {
            // First element must be exactly 1
            if(i == 0) {
                arr[0] = 1;
            } 
            // If the absolute difference is greater than 1, reduce the current element
            else if(abs(arr[i] - arr[i - 1]) > 1) {
                arr[i] = arr[i - 1] + 1;
            }

            // Keep track of the maximum element found so far
            maxEl = max(maxEl, arr[i]);
        }

        return maxEl;
    }
};

/*
*
* Dry Run
* Input: arr = [100, 1, 1000]
* 
* 1. Sort the array:
*    arr becomes [1, 100, 1000]
* 
* 2. Initialize maxEl = -1
* 
* 3. Loop iterations:
*    i = 0:
*      arr[0] is set to 1.
*      maxEl = max(-1, 1) = 1
*      arr state: [1, 100, 1000]
* 
*    i = 1:
*      abs(arr[1] - arr[0]) -> abs(100 - 1) = 99 > 1
*      Reduce arr[1] to arr[0] + 1 -> 1 + 1 = 2
*      maxEl = max(1, 2) = 2
*      arr state: [1, 2, 1000]
* 
*    i = 2:
*      abs(arr[2] - arr[1]) -> abs(1000 - 2) = 998 > 1
*      Reduce arr[2] to arr[1] + 1 -> 2 + 1 = 3
*      maxEl = max(2, 3) = 3
*      arr state: [1, 2, 3]
* 
* 4. Return maxEl: 3
*
*/
```

## 📌 Notes
* The problem statement explicitly states that we can *decrease* the value of any element, but we cannot *increase* it. This dictates why `arr[i] = arr[i - 1] + 1` is safe to do when `arr[i]` is too large, but we do nothing if `arr[i]` is equal to or smaller than `arr[i-1] + 1`.

## 🚀 Further Exploration
While the `O(N log N)` sorting approach is intuitive and fast enough, it is possible to solve this problem in `O(N)` time without sorting by using an array to count frequencies and tracking the maximum possible constraints. If you want an extra challenge, try to derive the `O(N)` logic!
