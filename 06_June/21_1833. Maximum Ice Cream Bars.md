## 📝 Problem Statement
It is a sweltering summer day, and a boy wants to buy some ice cream bars. 

At the store, there are `n` ice cream bars. You are given an array `costs` of length `n`, where `costs[i]` is the price of the `i-th` ice cream bar in coins. The boy initially has `coins` coins to spend, and he wants to buy as many ice cream bars as possible. 

**Note:** The boy can buy the ice cream bars in any order.

Return *the maximum number of ice cream bars the boy can buy with `coins` coins.*

> You must solve the problem by counting sort (or any equivalent efficient sorting approach).

---

## 🚀 Constraints

- `costs.length == n`
- `1 <= n <= 10^5`
- `1 <= costs[i] <= 10^5`
- `1 <= coins <= 10^8`

## 🧠 Approach

This problem acts as a perfect introduction to the **Greedy Strategy**. 
The goal is to maximize the *count* of ice cream bars purchased. To buy the maximum number of items with a limited budget, the most optimal and logical strategy is to always buy the cheapest available items first. 

**Step-by-Step Breakdown:**
1. **Sort the Array:** Sort the `costs` array in ascending order. This guarantees that we examine the cheapest ice creams first.
2. **Iterate & Greedily Pick:** Loop through the sorted array.
3. **Budget Check:** For each ice cream, if its cost is less than or equal to the remaining `coins`:
   - Increment our bought count.
   - Deduct the item's cost from our `coins` balance.
4. **Early Exit:** Because the array is sorted, the moment we encounter an ice cream that costs more than our remaining budget, we can immediately stop evaluating. Every subsequent ice cream will be equally or more expensive.
5. **Return Result:** Output the final tally of purchased ice creams.

---

## 🔑 Key Insights

- **Greedy is Optimal Here:** The problem only asks for the *maximum quantity*, not specific combinations of prices. There is no scenario where buying a more expensive ice cream allows you to buy *more* overall ice creams. Thus, a purely greedy choice (cheapest first) is mathematically optimal.
- **Early Termination:** Sorting doesn't just help pick the cheapest items; it allows us to break out of the loop early, preventing unnecessary iterations once our budget is exhausted.

## 💻 Solution Code

### Optimized Solution (C++)

```cpp
// Intuition: To maximize the number of ice cream bars we can buy, we should always pick the cheapest ones first. This is a classic Greedy approach.
// Approach: 
// 1. Sort the given 'costs' array in ascending order so the cheapest ice creams come first.
// 2. Iterate through the sorted array.
// 3. For each ice cream, check if we have enough 'coins' to buy it.
// 4. If yes, increment the count of bought ice creams and subtract its cost from our coins.
// 5. If not, since the array is sorted, we can't afford any of the remaining ice creams either, so we break/return early.
// Time Complexity: O(n log n) - Sorting the costs array takes O(n log n) time, where n is the number of elements. The iteration takes O(n) time.
// Space Complexity: O(1) - We only use a few integer variables (ignoring the internal recursion stack space used by the sorting algorithm).

// Approach-1 (Normal Sorting and greedily choosing)
class Solution {
public:
    int maxIceCream(vector<int>& costs, int coins) {
        // Sort costs to comfortably apply the greedy approach
        sort(begin(costs), end(costs));
        
        int count = 0; // Tracks the number of ice creams bought
        
        for(int &cost : costs) {
            // If current ice cream is too expensive, we stop
            if(cost > coins)
                return count;
            else {
                // Buy the ice cream and reduce our balance
                count++;
                coins -= cost;
            }
        }
        
        // Return total bought if we could afford all of them
        return count;
    }
};

/*
*
* Dry Run
* * Input: costs = [1, 3, 2, 4, 1], coins = 7
* * Step 1: Sort the costs array
* Sorted costs: [1, 1, 2, 3, 4]
* * Step 2: Iterate through the sorted array
* - cost = 1: 1 <= 7 (True) -> count = 1, coins = 7 - 1 = 6
* - cost = 1: 1 <= 6 (True) -> count = 2, coins = 6 - 1 = 5
* - cost = 2: 2 <= 5 (True) -> count = 3, coins = 5 - 2 = 3
* - cost = 3: 3 <= 3 (True) -> count = 4, coins = 3 - 3 = 0
* - cost = 4: 4 > 0 (False) -> Stop and return count
* * Output: 4
*
*/
```

## 📝 Notes

- **Alternative Approach (Counting Sort):** Since the maximum constraint for `costs[i]` is up to `10^5`, an alternative approach is to use **Counting Sort**. Instead of an `O(N log N)` comparison-based sort, you can allocate a frequency array of size `10^5 + 1` and populate it in `O(N)` time. Iterating through the frequency array takes `O(max_cost)` time, resulting in an overall Time Complexity of `O(N + max_cost)` and Space Complexity of `O(max_cost)`. This is highly beneficial if `N` is extremely large but the maximum price constraint is bounded.
