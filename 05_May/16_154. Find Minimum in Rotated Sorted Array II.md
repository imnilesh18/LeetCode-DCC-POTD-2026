## 📝 Problem Statement

Suppose an array of length `n` sorted in ascending order is rotated between `1` and `n` times. For example, the array `nums = [0,1,4,4,5,6,7]` might become:
* `[4,5,6,7,0,1,4]` if it was rotated `4` times.
* `[0,1,4,4,5,6,7]` if it was rotated `7` times.

Notice that rotating an array `[a[0], a[1], a[2], ..., a[n-1]]` 1 time results in the array `[a[n-1], a[0], a[1], a[2], ..., a[n-2]]`.

Given the sorted rotated array `nums` that may contain **duplicates**, return the minimum element of this array.

You must decrease the overall operation steps as much as possible.

---

## 🛑 Constraints

* `n == nums.length`
* `1 <= n <= 5000`
* `-5000 <= nums[i] <= 5000`
* `nums` is sorted and rotated between `1` and `n` times.

## 🧠 Approach

### The Challenge of Duplicates
This problem is an exact replica of "Find Minimum in Rotated Sorted Array", but with one major catch: **duplicates**. 
In standard binary search, we compare `nums[mid]` with `nums[r]`. If `nums[mid] > nums[r]`, we know the right half is unsorted and contains the minimum. If it's smaller, the left half contains the minimum. 

However, when duplicates are introduced (e.g., `nums = [3, 1, 3, 3, 3]`), we can run into a situation where `nums[l] == nums[mid] == nums[r]`. In this state, it is impossible to determine whether to search in the left half or the right half.

### The Solution: Duplicate Elimination
To counter this, we can safely shrink our search space. If we notice that the elements at our left and right pointers are identical to their adjacent elements, we simply skip them (`l++` and `r--`) until the duplicates are bypassed. Once the duplicates are eliminated from the boundaries, we can resume our standard binary search logic to locate the minimum element.

---

## 💻 Solution Code (Brute Force & Optimized)

```cpp
// --- BRUTE FORCE SOLUTION ---
// Intuition: The simplest way to find the minimum element in any array is to iterate through it completely.
// Approach: Traverse the array linearly from start to end, keeping track of the minimum value seen so far.
// Time Complexity: O(n) as we visit every single element exactly once.
// Space Complexity: O(1) since no extra space is used.

class SolutionBruteForce {
public:
    int findMin(vector<int>& nums) {
        int minVal = nums[0];
        for(int num : nums) {
            if(num < minVal) {
                minVal = num;
            }
        }
        return minVal;
    }
};

// --- OPTIMIZED SOLUTION ---
// Intuition: Since the array is sorted (but rotated), Binary Search is the optimal choice. To handle duplicates that break the binary search logic, we can proactively skip them at the boundaries before calculating the mid-point.
// Approach: 
// 1. Initialize left (l) and right (r) pointers.
// 2. Loop while l <= r. Shrink the search space by bypassing duplicates at both ends (l++ and r--).
// 3. Find the mid-point. Track the minimum element found so far in `resultIdx`.
// 4. If nums[mid] > nums[r], the minimum must be in the right half (l = mid + 1).
// 5. Otherwise, the minimum is in the left half (r = mid - 1).
// Time Complexity: O(log n) on average. However, in the worst-case scenario (e.g., all identical elements like [2,2,2,2]), the duplicate skipping takes O(n) time.
// Space Complexity: O(1) as we are only allocating space for a few pointer variables.

class Solution {
public:
    int findMin(vector<int>& nums) {
        int l = 0;
        int r = nums.size() - 1;

        int resultIdx = 0;

        while(l <= r) {

            // remove duplicates from left
            while(l < r && nums[l] == nums[l + 1])
                l++;

            // remove duplicates from right
            while(l < r && nums[r] == nums[r - 1])
                r--;

            int mid = l + (r - l) / 2;

            // update answer
            if(nums[mid] < nums[resultIdx])
                resultIdx = mid;

            // minimum lies on right side
            if(nums[mid] > nums[r]) {
                l = mid + 1;
            }
            // minimum can be on left side
            else {
                r = mid - 1;
            }
        }

        return nums[resultIdx];
    }
};

/*
*
* Dry Run
* Input: nums = [2, 2, 2, 0, 1]
* 
* Initialization: 
* l = 0, r = 4, resultIdx = 0
* 
* Iteration 1:
* l=0, r=4. 
* Left duplicates check: nums[0] == nums[1] (2==2). l becomes 1. 
* nums[1] == nums[2] (2==2). l becomes 2.
* Right duplicates check: nums[4] == nums[3] (1==0) is False.
* Now, l = 2, r = 4.
* mid = 2 + (4-2)/2 = 3.
* nums[mid] = nums[3] = 0. nums[resultIdx] = nums[0] = 2.
* Since 0 < 2, resultIdx updates to 3.
* nums[mid] > nums[r] (0 > 1) is False. 
* So, r = mid - 1 = 2.
* 
* Iteration 2:
* l = 2, r = 2.
* Duplicate checks skip because l is not strictly less than r.
* mid = 2 + (2-2)/2 = 2.
* nums[mid] = nums[2] = 2. nums[resultIdx] = nums[3] = 0.
* Since 2 < 0 is False, resultIdx remains 3.
* nums[mid] > nums[r] (2 > 2) is False.
* So, r = mid - 1 = 1.
* 
* Iteration 3:
* l = 2, r = 1. 
* Loop condition (l <= r) evaluates to False. Terminate.
* 
* Output: Return nums[resultIdx] which is nums[3] -> 0.
*
*/
```

## 📌 Notes
*   **Follow up Answer:** The presence of duplicates drastically changes the runtime complexity. While a standard Rotated Sorted Array search runs in strict $O(\log n)$ time, the worst-case scenario here forces us to linearly scan and skip elements, degrading the time complexity to $O(n)$.
