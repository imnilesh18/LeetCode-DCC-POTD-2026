## 📝 Problem Statement

You are given an integer array `nums` of length `n`.

Assume `arrk` to be an array obtained by rotating `nums` by `k` positions clock-wise. We define the **rotation function** `F` on `nums` as follows:

`F(k) = 0 * arrk[0] + 1 * arrk[1] + ... + (n - 1) * arrk[n - 1].`

Return the maximum value of `F(0), F(1), ..., F(n-1)`.

The test cases are generated so that the answer fits in a 32-bit integer.

### 🎯 Constraints:
* `n == nums.length`
* `1 <= n <= 10^5`
* `-100 <= nums[i] <= 100`

## 🧠 Approach

### Brute Force (Simulation)
The most intuitive approach is to physically rotate the array step-by-step or simulate the rotation and calculate $F(k)$ using a nested loop. However, since $n$ can be up to $10^5$, an $O(N^2)$ solution will definitely result in a **Time Limit Exceeded (TLE)** error. 

### Mathematical Optimization (O(N) Time)
Instead of relying on standard algorithms like Sliding Window or Dynamic Programming (which don't fit here because we need all elements and orientations change entirely), we can look for a mathematical pattern between $F(k)$ and $F(k+1)$.

Let's observe how the components of the sum change when moving from $F(0)$ to $F(1)$ for the array `[4, 3, 2, 6]`:
* **F(0)**: `(0*4) + (1*3) + (2*2) + (3*6)`
* **F(1)**: `(1*4) + (2*3) + (3*2) + (0*6)`

Notice the contribution changes:
* The multiplier for `4` increases by 1 (from 0 to 1).
* The multiplier for `3` increases by 1 (from 1 to 2).
* The multiplier for `2` increases by 1 (from 2 to 3).
* The multiplier for `6` drops from 3 (which is $n-1$) to 0.

If we add the total sum of the array (`4 + 3 + 2 + 6`) to $F(0)$, every element gets an extra $+1$ multiplier. However, the last element (`6`) should have dropped to $0$ instead of increasing to $4$ ($n$). To correct this, we subtract $n \times 6$ from the total. 

**General Formula:**
$F(k) = F(k-1) + \text{sum} - n \times \text{nums}[n - k]$

This means if we calculate the total sum of the array and $F(0)$ initially, we can calculate any subsequent $F(k)$ in $O(1)$ time!

## 💻 Solution Code
```cpp
// Intuition: A brute-force simulation of rotations takes O(N^2) and results in TLE. By analyzing the difference between F(k) and F(k+1), we can derive a mathematical relationship to compute the next state in O(1) time.
// Approach: 
// 1. Calculate the array's total sum and the initial F(0).
// 2. Iteratively calculate the next F(k) using the derived formula: newF = previousF + sum - n * last_element_of_current_rotation.
// 3. Keep track of the maximum value encountered.
// Time Complexity: O(n) as we iterate through the array sequentially.
// Space Complexity: O(1) since we only use a few constant variables for tracking sums and results.
class Solution {
public:
    int maxRotateFunction(vector<int>& nums) {
        int n = nums.size(); // Get the size of the array

        int sum = 0;
        int F = 0;

        for(int i = 0; i < n; i++) {
            sum += nums[i]; // Calculate total sum of array
            F += i*nums[i]; // Calculate initial F(0)
        }

        int result = F; // Initialize result with F(0)

        for(int k = 0; k <= n-1; k++) {
            int newF = F + sum - n*nums[n-1-k]; // Compute next F using math derivation

            result = max(result, newF); // Update maximum value found

            F = newF; // Store current F for the next iteration
        }

        return result;
    }
};

/*
*
* Dry Run
* Array: nums = [4, 3, 2, 6], n = 4
* Initial sum = 4 + 3 + 2 + 6 = 15
* Initial F(0) = (0*4) + (1*3) + (2*2) + (3*6) = 25
* result = 25
* 
* Iteration k = 0:
* newF = 25 + 15 - 4 * nums[3] = 40 - 24 = 16
* result = max(25, 16) = 25
* F = 16
* 
* Iteration k = 1:
* newF = 16 + 15 - 4 * nums[2] = 31 - 8 = 23
* result = max(25, 23) = 25
* F = 23
* 
* Iteration k = 2:
* newF = 23 + 15 - 4 * nums[1] = 38 - 12 = 26
* result = max(25, 26) = 26
* F = 26
* 
* Iteration k = 3:
* newF = 26 + 15 - 4 * nums[0] = 41 - 16 = 25
* result = max(26, 25) = 26
* F = 25
* 
* Final Result: 26
*
*/
```

## 🔍 Key Insights
1. **Recognize the pattern:** Often, problems involving array rotations or shifts hide mathematical relations between consecutive states. 
2. **Avoid literal simulation:** If the constraints are $10^5$, physical array manipulation will fail. Look for prefix sums, sliding windows, or recurrence relations.
3. **Tracking the "Last" Element:** When shifting right virtually, the element that drops from index `n-1` to `0` changes relative to `k`. The index of this element in the original array is `n - 1 - k`.
