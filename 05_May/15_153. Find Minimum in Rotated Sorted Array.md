## 📝 Problem Statement

Suppose an array of length `n` sorted in ascending order is rotated between `1` and `n` times. For example, the array `nums = [0,1,2,4,5,6,7]` might become:
* `[4,5,6,7,0,1,2]` if it was rotated 4 times.
* `[0,1,2,4,5,6,7]` if it was rotated 7 times.

Notice that rotating an array `[a[0], a[1], a[2], ..., a[n-1]]` 1 time results in the array `[a[n-1], a[0], a[1], a[2], ..., a[n-2]]`.

Given the sorted rotated array `nums` of **unique** elements, return the minimum element of this array.

*You must write an algorithm that runs in `O(log n)` time.*

### 🎯 Constraints:
* `n == nums.length`
* `1 <= n <= 5000`
* `-5000 <= nums[i] <= 5000`
* All the integers of `nums` are **unique**.
* `nums` is sorted and rotated between `1` and `n` times.

## 🔍 Approach & Explanation

The problem explicitly asks for an $O(\log n)$ time complexity, which strongly suggests that we need to use **Binary Search**.

1.  **Understanding Rotated Sorted Arrays:**
    In a standard sorted array, any element is strictly smaller than the elements to its right. However, in a rotated sorted array, the elements increase up to a certain point (the rotation point) and then abruptly drop to the absolute minimum before increasing again.
2.  **Binary Search Strategy:**
    We can set two pointers, `l` (left) at `0` and `r` (right) at `n - 1`. We calculate the middle index `mid`.
3.  **Determining the Search Space:**
    We compare the middle element (`nums[mid]`) with the rightmost element (`nums[r]`):
    *   **Case 1: `nums[mid] > nums[r]`**
        This means the rotation pivot (and therefore the minimum element) lies somewhere in the right half of the array. The `mid` element itself is greater than the rightmost element, so it cannot be the minimum. Therefore, we discard the left half and update `l = mid + 1`.
    *   **Case 2: `nums[mid] <= nums[r]`**
        This indicates that the right half of the array is strictly sorted in increasing order. Because it's properly sorted, the minimum element cannot lie strictly to the right of `mid`. It could be `mid` itself, or it could be in the left half. Therefore, we update `r = mid`.
4.  **Loop Termination:**
    We use the condition `while(l < r)`. If we were to use `l <= r`, we would risk entering an infinite loop because updating `r = mid` when `l == r` leaves the pointers completely unchanged. The loop naturally terminates when `l == r`, at which point both pointers land on the minimum element.

## 💻 Solution Code

<details open>
<summary><b>C++ Solution (Optimized Binary Search)</b></summary>

```cpp
// Intuition: The array is sorted but rotated. The O(log n) requirement points to Binary Search. By comparing the middle element to the rightmost element, we can determine which half of the array contains the pivot (minimum element).
// Approach: 
// 1. Initialize left `l` and right `r` pointers.
// 2. Loop while `l < r`. Calculate `mid`.
// 3. If `nums[mid] > nums[r]`, the right side is unsorted, meaning the minimum must be to the right of `mid`. Set `l = mid + 1`.
// 4. Otherwise, the right side is sorted, meaning the minimum is at `mid` or to its left. Set `r = mid`.
// 5. When `l` and `r` converge, they point to the minimum element.
// Time Complexity: O(log n) - standard binary search dividing the search space by half.
// Space Complexity: O(1) - only using a few variables for pointers.

class Solution {
public:
    int findMin(vector<int>& nums) {
        int n = nums.size();
        
        int l = 0, r = n - 1; // Pointers for binary search
        
        while(l < r) {
            // Find mid, prevent integer overflow
            int mid = l + (r - l) / 2;
            
            // If mid element is greater than rightmost element
            if(nums[mid] > nums[r]) {
                // Minimum must be in the right half
                l = mid + 1;
            } else {
                // Minimum is at mid or in the left half
                r = mid;
            }
        }
        // When l == r, we've found the minimum element
        return nums[r];
    }
};

/*
*
* Dry Run
* 
* Input: nums = [3, 4, 5, 1, 2]
* 
* Iteration 1:
* l = 0, r = 4. mid = 0 + (4 - 0) / 2 = 2.
* nums[mid] = nums[2] = 5. nums[r] = nums[4] = 2.
* Since 5 > 2 is true: l = mid + 1 = 3.
* 
* Iteration 2:
* l = 3, r = 4. mid = 3 + (4 - 3) / 2 = 3.
* nums[mid] = nums[3] = 1. nums[r] = nums[4] = 2.
* Since 1 > 2 is false: r = mid = 3.
* 
* Loop condition: l < r -> 3 < 3 is False.
* 
* Return: nums[r] -> nums[3] -> 1.
*
*/
```

## 🧠 Key Insights

* **Why Compare with `nums[r]`?** Comparing `nums[mid]` with `nums[l]` can be ambiguous when a sub-array is strictly increasing and hasn't been rotated within that window. However, `nums[r]` acts as a perfect baseline; if `nums[mid]` is strictly larger than the right boundary, the drop-off (minimum) *must* happen after `mid`. 
* **Avoiding Infinite Loops:** Using `r = mid` necessitates the loop condition `while (l < r)`. If we used `while (l <= r)`, updating `r = mid` when `l == r` would trap the algorithm endlessly.
