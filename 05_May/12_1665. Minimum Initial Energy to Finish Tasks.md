# 1665. Minimum Initial Energy to Finish Tasks

**Problem Link:** [LeetCode 1665. Minimum Initial Energy to Finish Tasks](https://leetcode.com/problems/minimum-initial-energy-to-finish-tasks/)
**Difficulty:** Hard
**Tags:** Array, Greedy, Sorting, Binary Search



## 📝 Problem Statement

You are given an array `tasks` where `tasks[i] = [actual_i, minimum_i]`:
*   `actual_i` is the actual amount of energy you spend to finish the `i`th task.
*   `minimum_i` is the minimum amount of energy you require to begin the `i`th task.

For example, if the task is `[10, 12]` and your current energy is `11`, you cannot start this task. However, if your current energy is `13`, you can complete this task, and your energy will be `3` after finishing it.

You can finish the tasks in **any order** you like.

Return the **minimum** initial amount of energy you will need to finish all the tasks.

---

## 🚦 Constraints

*   `1 <= tasks.length <= 10^5`
*   `1 <= actual_i <= minimum_i <= 10^4`

## 🧠 Approach

The problem asks for the *minimum* starting energy to fulfill a sequence of operations. This requires two major insights:

<details>
<summary><b>Insight 1: Monotonicity (Binary Search on Answer)</b></summary>
If you can successfully complete all tasks with an initial energy of `X`, you can definitely complete them with `X + 1`, `X + 2`, etc. Conversely, if you fail with energy `X`, you will definitely fail with `X - 1`. This monotonic behavior indicates that we can binary search our answer.
The lowest possible energy is `0`. The highest possible energy (worst-case) is `10^5` tasks $\times$ `10^4` max energy = `10^9`.
</details>

<details>
<summary><b>Insight 2: Optimal Execution Order (Greedy)</b></summary>
Even with Binary Search, the order in which we attempt the tasks matters. To maximize our chances of completing all tasks, we should perform tasks that leave us with the highest "residual" or "leftover" energy first. 
The leftover energy after starting a task is `minimum - actual`. A task with a high initial requirement but low actual cost should be prioritized. It acts as a heavy "gate" that we should pass through while our energy is still high, saving our remaining pool of energy for subsequent tasks. Therefore, we sort the tasks in descending order of `(minimum - actual)`.
</details>

### Brute Force Solution (Concept)
A naive approach would be to try every single permutation of the tasks ($N!$ permutations) and for each permutation, simulate the process to find the minimum starting energy. Given $N = 10^5$, $O(N!)$ will yield a Time Limit Exceeded (TLE) error immediately.

### Optimized Solution
We combine **Greedy Sorting** with **Binary Search on Answer**. We first sort the tasks by their residual energy difference. Then, we apply binary search between `l = 0` and `r = 1e9`, validating if a `mid` value of energy is sufficient to complete the optimally sorted tasks.

## 💻 Solution Code

```cpp
// Intuition: If energy E works, E+1 works. If E fails, E-1 fails (Monotonic property -> Binary Search). Also, to maximize our chances, we should tackle tasks that leave us with the most "residual" energy first.
// Approach: 
// 1. Sort the tasks in descending order based on the difference between minimum required energy and actual energy consumed (minimum - actual).
// 2. Apply Binary Search on the answer. Set 'l' to 0 and 'r' to 1e9 (max possible energy based on constraints).
// 3. For each 'mid' in binary search, use an 'isPossible' function to check if we can complete all tasks sequentially from the sorted array.
// 4. If 'mid' works, store it as a potential result and search the left half (r = mid - 1) for a strictly smaller valid energy. Otherwise, search the right half (l = mid + 1).
// Time Complexity: O(N log N) dominated by the sorting step. The Binary Search takes O(log(1e9)) iterations, doing an O(N) check each time, which acts as O(N) practically.
// Space Complexity: O(1) auxiliary space as we sort the array in-place and use only a few integer variables for pointers.

class Solution {
public:
    bool isPossible(vector<vector<int>>& tasks, int mid) {
        for(auto &task : tasks) {
            int actual  = task[0];
            int minimum = task[1];

            // If current energy is less than the minimum required, we can't start this task
            if(minimum > mid) {
                return false;
            }

            // Deduct the actual energy spent on this task
            mid -= actual;
        }
        
        // Successfully completed all tasks
        return true;
    }

    int minimumEffort(vector<vector<int>>& tasks) {
        int n = tasks.size();

        // Search boundaries based on problem constraints
        int l = 0;
        int r = 1e9; 

        int result = INT_MAX;

        // Custom comparator to sort by (minimum - actual) descending
        auto lambda = [](auto &task1, auto& task2) {
            int diff1 = task1[1] - task1[0];
            int diff2 = task2[1] - task2[0];

            return diff1 > diff2;
        };

        sort(begin(tasks), end(tasks), lambda);

        // Binary Search for the minimum initial energy
        while(l <= r) {
            int mid = l + (r - l) / 2;

            if(isPossible(tasks, mid)) {
                result = mid; // Update result with current valid energy
                r = mid - 1;  // Try to find a strictly smaller valid energy
            } else {
                l = mid + 1;  // Energy was insufficient, we need more
            }
        }

        return result;
    }
};

/*
*
* Dry Run
* 
* Input: tasks = [[1,2], [2,4], [4,8]]
* 
* 1. Calculate Differences (min - actual):
*    [1,2] -> Diff = 1
*    [2,4] -> Diff = 2
*    [4,8] -> Diff = 4
* 
* 2. Sort by difference descending: 
*    tasks = [[4,8], [2,4], [1,2]]
* 
* 3. Binary Search Setup: l = 0, r = 1000000000
*    Let's fast forward the binary search to where mid = 8.
*    
*    isPossible(tasks, 8):
*    - Task [4,8]: min 8 <= 8 (True). Remaining mid = 8 - 4 = 4.
*    - Task [2,4]: min 4 <= 4 (True). Remaining mid = 4 - 2 = 2.
*    - Task [1,2]: min 2 <= 2 (True). Remaining mid = 2 - 1 = 1.
*    Returns True. Result updated to 8. Search moves left (r = 7).
* 
*    Next iteration, let's check mid = 7.
*    isPossible(tasks, 7):
*    - Task [4,8]: min 8 <= 7 (False).
*    Returns False. Search moves right (l = 8).
* 
* 4. Loop terminates. Result returned is 8.
*
*/
```

## 📌 Notes & Key Insights

*   **Why not sort by maximum `minimum` energy directly?** If a task requires `100` energy but consumes `99`, it leaves you with just `1` energy. A task that requires `55` but consumes `5` leaves you with `50` energy. Prioritizing the one that leaves the most energy allows you to chain more tasks together sequentially.
*   **Alternative Greedy Approach**: It is actually possible to solve this in $O(N \log N)$ purely through greedy math without the binary search layer by iterating backward or intelligently accumulating the energy required, which serves as a great further exploration challenge!
